# Feature: Wanted Semantic Embeddings (Voyage voyage-3)

## Metadata

issue_number: `wanted-emb`
adw_id: `1`
issue_json: `docs/superpowers/specs/2026-06-23-wanted-embeddings-design.md`

## Feature Description

Add semantic retrieval to the Wanted Matcher by plugging a second
`CandidateRetriever` implementation behind the existing seam. Embeddings are
generated with **Voyage `voyage-3` (1024 dims)** — Anthropic's recommended
embeddings provider — stored in pgvector columns, and combined with the existing
full-text-search (FTS) retrieval via Reciprocal Rank Fusion (RRF). The Haiku
re-rank stage and `phase-a`/`phase-b`/`run` orchestration are unchanged.

This lifts Decision **D1** (embeddings deferred) from
`docs/superpowers/specs/2026-06-01-wanted-adaptation-design.md`. The approved
design contract is `docs/superpowers/specs/2026-06-23-wanted-embeddings-design.md`
— **implement it exactly**; do not substitute providers, dimensions, or fusion
strategy.

## User Story

As a **seeker posting a brief**
I want **the Matcher to find apps and builders whose meaning matches my need**,
not just keyword overlap
So that **paraphrased needs (e.g. "recurring billing" ≈ "subscription management")
surface the right matches that FTS alone would miss.**

## Problem Statement

The v1 Matcher retrieves candidates with Postgres FTS over `apps.search_vector`
plus hard filters, then re-ranks with Haiku. FTS is purely lexical: a brief that
describes a need in different words than an app's listing won't retrieve it, even
when they are semantically identical. There is no semantic retrieval arm.

## Solution Statement

Implement a `SemanticCandidateRetriever` that runs **both** the existing FTS arm
and a new vector-cosine arm (over Voyage embeddings stored in pgvector) inside the
same hard-filtered candidate set, fuses their rankings with RRF, and returns the
top-N to the unchanged Haiku re-rank. Embeddings are computed best-effort on write
(briefs/apps) and by a nightly Vercel Cron (builder capability vectors + NULL
backfill). The matcher selects the semantic retriever when `VOYAGE_API_KEY` is set
and otherwise falls back to the existing `FtsCandidateRetriever`, so production is
safe before the key is configured and degrades to FTS whenever an embedding is
missing.

### 1. Migration 0040 + cosine RPCs + types

- **Task ID**: migration-and-types
- **Depends On**: none
- **Assigned To**: db-engineer
- **Agent Type**: `db-agent`
- **Parallel**: true
- **Owns Files**: `packages/db/migrations/0040_wanted_embeddings.sql`,

- **Context**: Author an idempotent migration following the style of
  `packages/db/migrations/0031_briefs.sql` and `0034_extend_profiles_apps.sql`
  (snake_case, `if not exists`, down-notes comment block). It must:
  (a) `create extension if not exists vector;`
  (b) `alter table public.briefs add column if not exists embedding vector(1024);`
  (c) `alter table public.apps add column if not exists embedding vector(1024);`
  (d) `alter table public.profiles add column if not exists capability_embedding vector(1024);`
  (e) HNSW cosine indexes on each of the three columns (`using hnsw (<col> vector_cosine_ops)`).
  (f) `create or replace function public.match_apps_by_embedding(query_embedding vector(1024),
    match_count int default 30, exclude_saas boolean default false, oss_only boolean default false)
    returns table(id uuid, distance real)` — selects from `public.apps` where
  `is_published = true and embedding is not null`, applies the licensing pre-filter
  (when `exclude_saas`: `licensing is null or licensing <> 'saas'`; when `oss_only`:
  `licensing is null or licensing = 'oss'`), orders by `embedding <=> query_embedding`,
  limits to `match_count`. `language sql stable security invoker`.
  (g) `create or replace function public.match_builders_by_embedding(query_embedding vector(1024),
    match_count int default 50) returns table(id uuid, distance real)` — selects from
  `public.profiles` where `accepts_requests = true and capability_embedding is not null`,
  orders by `capability_embedding <=> query_embedding`, limits to `match_count`.
  `language sql stable security invoker`. (Remaining structured builder filters stay in TS.)
  The new columns inherit each table's existing RLS — no new policies needed (matcher/cron
  use the service-role admin client). Apply via `mcp__supabase__apply_migration`, then run
  `mcp__supabase__get_advisors` (expect no new security errors), then regenerate
  `apps/web/lib/supabase/types.ts` via `mcp__supabase__generate_typescript_types`.
- **Actions**:

### 4. SemanticCandidateRetriever + RRF + run.ts selection

- **Task ID**: semantic-retriever
- **Depends On**: migration-and-types
- **Assigned To**: retriever-engineer
- **Agent Type**: `general-purpose`
- **Parallel**: true
- **Owns Files**: `apps/web/lib/wanted/matching/semantic-retriever.ts`,
  `apps/web/lib/wanted/matching/rrf.ts`,
  `apps/web/lib/wanted/matching/run.ts`,
  `apps/web/lib/wanted/matching/semantic-retriever.test.ts`,
  `apps/web/lib/wanted/matching/rrf.test.ts`
- **Context**: `rrf.ts` — a pure `reciprocalRankFusion(lists: string[][], k = 60):
string[]` that takes ranked id lists and returns the fused id ranking
  (`score = Σ 1/(k + rank_i)`, descending). `semantic-retriever.ts` —
  `SemanticCandidateRetriever implements CandidateRetriever`
  (interface in `./retriever`). Constructor takes `(client: SupabaseClient<Database>,
briefEmbedding: number[] | null)`. **retrieveApps**: get the FTS-ranked, hard-filtered
  candidates by delegating to `FtsCandidateRetriever.retrieveApps` (reuse — do not
  duplicate filters). If `briefEmbedding` is non-null, call
  `client.rpc('match_apps_by_embedding', { query_embedding, match_count, exclude_saas,
oss_only })` (map licensing the same way the FTS retriever does) to get the vector-ranked
  ids; RRF-fuse the FTS id order with the vector id order; materialize the union of
  candidates (the FTS arm already has full rows; fetch any vector-only ids via one
  `.in('id', ...)` query, applying `is_published` + the existingStack drop) and return in
  fused order, capped at `k`. If `briefEmbedding` is null, return exactly the FTS result
  (degrade to FTS-only). **retrieveBuilders**: same shape over
  `match_builders_by_embedding` (capability cosine), fused with the FTS retriever's
  structured candidate ordering; null embedding → structured-only. `run.ts`: add
  `createDefaultRetriever(admin, briefEmbedding)` returning `SemanticCandidateRetriever`
  when `process.env.VOYAGE_API_KEY` is set else `FtsCandidateRetriever`; extend the brief
  `select('content')` to `select('content, embedding')`; replace
  `args.retriever ?? createFtsRetriever(admin)` with
  `args.retriever ?? createDefaultRetriever(admin, briefRow.embedding)` (move retriever
  construction to after the brief row is read). Keep `phase-a`/`phase-b` calls unchanged.
  Tests: faked Supabase client (canned rows + a `.rpc` stub returning canned id/distance);
  assert RRF order, that a null brief embedding yields the pure-FTS result, and that
  vector-only ids get materialized.
- **Actions**:
  - Implement `rrf.ts` (+ test), `semantic-retriever.ts` (+ test)
  - Wire `createDefaultRetriever` + brief-embedding read into `run.ts`
  - `pnpm test` and `pnpm typecheck`

## Testing Strategy

### Unit Tests

- `voyage.test.ts` — mocked `fetch`: request shape (url, `voyage-3`, `input_type`, batch,
  auth header), response parse, error/null path.
- `embed.test.ts` — recipes → Voyage call (mocked), best-effort null on failure.
- `capability.test.ts` — faked Supabase client + mocked embed: top-5 selection, NULL sweep,
  per-row failure isolation, counts.
- `rrf.test.ts` — pure fusion: known inputs → expected order; ties, single list, empty.
- `semantic-retriever.test.ts` — faked client + `.rpc` stub: RRF order, null-embedding →
  pure-FTS, vector-only id materialization, hard filters preserved.
- `route.test.ts` (cron) — 401 without secret; success invokes both batch functions.

### Edge Cases

- Voyage down on write → row saved with `embedding = NULL`, request still succeeds.
- Brief embedding NULL at match time → vector arm skipped, FTS-only result.
- `VOYAGE_API_KEY` unset → matcher uses `FtsCandidateRetriever`; writes skip embedding.
- Cron without valid `CRON_SECRET` → 401, no work.
- Builder with zero published apps → no capability vector, excluded from builder vector arm.
- Empty brief content (just created) → embedding text is near-empty; tolerated (cron/refine
  re-embeds once content materializes).

## Notes

- **No new npm dependency.** The Voyage client uses Node 22's global `fetch` against the
  Voyage REST API — no official SDK, consistent with "no new stack without approval" beyond
  the already-approved Voyage provider.
- **E4 (Vercel Cron) matches existing convention.** `apps/web/vercel.json` already defines a
  `crons` array and `/api/cron/*` routes are already `CRON_SECRET`-protected (`pick-featured`,
  `refresh-scores`). The README's "pg_cron handles cron" line is partially historical; Vercel
  Cron is in active use. This plan extends the existing pattern, not a new scheduler.
- **Retriever seam stays stable.** `phase-a.ts`/`phase-b.ts` are untouched; only `run.ts`
  changes (retriever selection + reading `briefs.embedding`). The semantic retriever composes
  `FtsCandidateRetriever` to reuse hard filters rather than duplicating them.
- **Brief embedding is `input_type: 'query'`** (stored at write time); apps and capabilities
  are `'document'`. Cosine compares a query embedding against document embeddings.
- **Database expert `no_pgvector` note is now obsolete** — Task 9 flags the self-improve.
